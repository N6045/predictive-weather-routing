--- Slide 9 ---
Proposed System
The user enters a start point , start time and destination in the web application.
The system generates a route on an interactive map.
The route geometry is sampled at key points such as origin, midpoint, and destination.
The application requests weather data for these coordinates.
Weather details are displayed beside the route together with simple warnings.
The first version acts as a decision-support tool rather than a full automatic rerouting engine.

Pseudocode- Road Mapping & Path Computation Algorithm
Start the algorithm.
Read the input start location text and destination location text.
Convert both location texts into geographic coordinates using geocoding.
Check if either the start or destination coordinates are null.
If any coordinate is null, return "Invalid start or destination" and terminate the algorithm.
Request the route from the OSRM service using the obtained coordinates.
Check if the route response is empty.
If the response is empty, return "Route not found" and terminate the algorithm.
Extract the route path coordinates, total distance, and total travel time from the response.
Compute the midpoint index as half of the length of the route path.
Determine the midpoint coordinate using the midpoint index.
Draw the route path on the map.
Return the route path, total distance, total travel time, and midpoint coordinate.
End the algorithm.

Pseudocode- Route–Weather Synchronisation & Timeline Generation Algorithm
Start the algorithm.
Read departure time, total travel time, and forecast data (start, midpoint, destination).
Compute arrival times:
Start = departure time
Midpoint = departure time + (total travel time / 2)
Destination = departure time + total travel time
For each location (start, midpoint, destination):
 a. Initialize minimum time difference to infinity and closest forecast to null.
 b. For each forecast entry, compute the absolute time difference from arrival time.
 c. If the difference is smaller, update the closest forecast.
 d. Store the closest forecast for that location.
Return matched forecasts for start, midpoint, and destination.
End the algorithm.

Pseudocode- Backend Orchestration & API Delivery Algorithm
Start the algorithm.
Read start location, destination location, and departure time.
Geocode locations to obtain coordinates.
If coordinates are invalid, return "Location not found" and terminate.
Request route from OSRM; if unavailable, return "Route unavailable" and terminate.
Extract route path, distance, and travel time; compute midpoint and its place name.
Fetch weather forecast data for start, midpoint, and destination.
Compute arrival times based on departure time and travel duration.
Match closest forecast entries for each location.
Analyze forecasts to generate weather warnings.
Construct and return structured route-weather response.
End the algorithm.

Pseudocode- Frontend Visualisation & UI Algorithm
Start the algorithm.
While the user types in the search box, fetch and display location suggestions after a short delay.
When the user selects start and destination and initiates the request, send the inputs to the backend.
Receive the route-weather response.

Clear previous UI elements.
Render the route on the map and adjust the view.
Display route summary details (locations, midpoint, distance, travel time).
Display weather information for start, midpoint, and destination.
If warnings exist, display them prominently.
End the algorithm.

🔹 API Endpoints

POST /api/history – Save route with timestamp
DELETE /api/history – Clear records (admin only)
GET /api/captcha – Generate CAPTCHA { id, svg }
POST /api/login – Validate user + CAPTCHA
POST /api/register – Register user (hashed password)

--- Slide 19 ---
CAPTCHA & Password Security


🔹 CAPTCHA System
Generated using svg-captcha (distorted SVG)
Returns { id, svg }; stored in memory (expires in 5 min)
Working:
 Generate → Display → User Input → Verify → Delete (one-time use)
Purpose:
Prevent bots (OCR-resistant distortions)
Ensure human verification
🔹 Password Handling
Frontend Validation:
≥ 8 characters, uppercase, lowercase, number, special character
Real-time strength check
Backend Security:
Hashed using bcryptjs (cost: 10)
Stored securely (no plain text)
Login Flow:
 CAPTCHA → Password check (bcrypt.compareSync()) → Access control
